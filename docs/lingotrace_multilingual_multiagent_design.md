# LingoTrace 升级方案设计白皮书：多语种与多 Agent 终端演进

当前的 LingoTrace 是一套高度自动化的日语学习与记忆管理工作流系统。如果要在未来将其升级为支持**多语种（如英语、法语、西班牙语）**，以及平滑适配如 **Antigravity、Claude Code、Trae SOLO** 等不同的 Agent 终端产品，我们需要在尊重现有架构的基础上，评估潜在的挑战与合理的演进路径。

以下是详细的挑战分析与重构方案设计。

---

## 模块一：多语种支持 (Multi-lingual Support) 演进方案

要支持多语种，最大的挑战在于**语言学特征的抽象化**。

### 1. 面临的挑战
- **发音与重音绑架**：当前系统强依赖本地离线词典（`fugashi`, `unidic-lite`）来生成 `accent_display`（如 `ことば②`）。其他语言没有“音调（Pitch Accent）”概念，而是依赖 IPA 音标、连读（法语 Liaison）或重音节标记。
- **正字法与变体耦合**：目前专门针对中日汉字设计了 `kanji_diff` 和 `kanji_diff_pairs` 字段。但如果是法语，痛点变成了阴阳性（Gender）和动词变位（Conjugation）；如果是英语，可能是美/英拼写差异。
- **命名空间与标签硬编码**：系统中的 Tag 被硬编码为 `jp/vocab`、`jp/grammar`。在源笔记生成时，给 ListenKit 的参数也被固定为 `--language Japanese`。

### 2. 核心改动设计
#### A. 元数据（Frontmatter）的 Schema 抽象化
将目前硬编码的卡片属性重构为**“基础通用属性 + 语种插件属性”**的结构：
- **基础属性 (通用)**：`headword`, `meaning`, `source_notes`, `review_stage`, `seen_count` 等 SRS 调度所必须的字段。
- **发音层接口 (Phonetic Layer)**：废弃 `reading` 和 `accent_display`，抽象为通用的 `pronunciation` 字段。
- **语言特征接口 (Language Features)**：废弃 `kanji_diff`，建立通用的 `variants` 或 `features` 字段，供各语种挂载自身独有特性（如词性变换、阴阳性）。

#### B. NLP 与词典引擎的抽象化
将底层的本地离线词典调用抽象为接口，根据执行时的目标语种加载对应的资源包。

#### C. 环境与标签的动态注入
增加全局配置读取机制，动态配置语言标识（例如将 `jp/` 前缀参数化为 `${tag_prefix}`），使得系统运行时可以动态适配各类标签路径。

## 模块二：多 Agent 产品适配演进方案 (Multi-Agent Support)

如果要将本系统运行在 Antigravity、Claude Code、Trae SOLO 等通用 Agent 产品中，我们要避免过早改动产品架构，不引入不必要的中间层。Obsidian CLI 本身是一套完全独立的命令行工具集，不仅限于 Codex，任何具备终端执行能力的 Agent 都可以自由调用。

### 1. 面临的挑战与适配策略
- **跨平台 Prompt 格式差异**：不同 Agent 产品对 System Prompt 的挂载方式不同（有的依赖特定文件后缀，有的依赖全局配置文件）。
  - **策略**：无需改变架构，只需将现有的 `SKILL.md` 指令逻辑通过简单的文件结构调整，映射为各平台推荐的提示词入口格式即可。
- **底层模型（LLM）能力的客观差异**：由于不同 Agent 产品背后的底层模型在多轮指令遵循、复杂状态管理、工具调用容错性上存在差异，原本在某一个模型下能严格执行的“先查后写”逻辑，在另一个模型下可能会出现遗漏。
  - **策略**：当前阶段接受这种能力差异，继续保持现有的 Obsidian CLI 调用方式。远期则通过进一步界定 LLM 与 Python 的分工来规避大模型幻觉带来的风险。

### 2. 最远期计划：LLM 与 Python 脚本的绝对分工边界

目前系统已经在很大程度上分离了“逻辑判断（LLM）”与“客观计算（脚本）”（例如 ASR 识别与 SRS 次日排期已完全交给工具）。为了在未来让不同能力的 Agent 都能极度稳定地运作，我们必须在远期计划中将两者的分工进行极其严谨的切割。

#### A. 必须保留给大模型（LLM）执行的任务
这些任务高度依赖语义理解、语境推理和模糊判定，是 LLM 不可替代的主场，必须且只能由大模型完成：
1. **听力句子的提纯与质量评估**：从大段泛听文本中，评估句子的实用性和结构，挑选出符合要求的“可直接背的常用句”，并梳理其骨架与场景。
2. **泛文本材料的知识结构化**：阅读无格式的文章或长视频转写，总结内容脉络，并提炼出高价值的语法和词汇清单。
3. **语义关联与图谱构建**：为词汇寻找合适的 `confusable_with` (混淆词)，为语法卡片寻找 `contrast_with`；依靠上下文理解用户错题的原因，并总结“下次怎么避免”的复盘建议。
4. **口语卡片场景研判**：评估一句话是否符合日常真实交流的语境，为其分配极其具体的 `Scene`（家庭、点餐、职场）和发话角色（`speaker_role`）。
5. **对话人物分离**：当底层的转写无法完美分离发言人时，凭借语境和对话逻辑标记 `A:` 与 `B:` 的轮替。

#### B. 必须下沉并完全交由 Python/Shell 脚本执行的任务
这些任务属于状态管理、算术计算与防重校验，涉及严格的条件分支。大模型在执行多级 If-Else 指令时极易出现步骤遗漏。以下逻辑必须全部封装为确定性的脚本供 Agent 调用：
1. **双层词汇库的路由与防重逻辑**：目前 Agent 需要遵照指令“先搜索 Focus Root，再搜索 Base Root”，并根据结果决定是创建新笔记还是合并到旧笔记。远期应将此逻辑封装在 Python 网关中，Agent 只负责输入提取好的“生词与词义”，由脚本执行真正的存取判断。
2. **字典缓存与重音查询**：Agent 严禁自主猜测重音，获取重音这一动作必须严格绑定在对脱机脚本（`setup_offline_dictionary.py` 等）的查询结果之上。
3. **SRS 的时间衰减与排期算法**：目前已由 `run-next-day-review-update.sh` 妥善管理。严格禁止 Agent 介入对 `next_review` 和 `review_stage` 跨度天数的计算与更新。
4. **汉字差异对的自动化对齐（针对多语种）**：如果涉及确定性的语言学特征转换（如繁简对比、词性变位校验表），应交由脚本通过规范化字典预处理，而不是消耗大模型的内部知识库或引发错误联想。
5. **最终结果的合法性校验**：卡片生成后的必填字段检查、音频链接的真实性验证等，目前已由验证脚本接管，需继续保持隔离，不可由大模型自我宣称“校验通过”。
