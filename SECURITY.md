# 安全政策

这个仓库是公开的日语学习系统模板与工具仓库。真实学习笔记、教材音频、图片、PDF、完整转写稿、Obsidian 本地状态和任何密钥都不应该进入公开仓库。

## 支持范围

安全维护范围是默认分支 `main` 上公开发布的文件，包括：

- 公开说明文档
- `codex-skills/` 下的技能说明与公开脚本
- `tools/` 下的公开工具
- `系统配置/` 下的公开模板与路径配置
- `学习系统/总训练.base` 这个空的训练表框架

私人学习资料、媒体文件、本地缓存和个人 Obsidian 配置不属于公开仓库内容。

## 不要提交的内容

请不要在 issue、pull request 或提交中加入：

- 个人学习笔记、日记、复习记录或课堂原始笔记
- 教材音频、视频、图片、PDF、扫描件或完整转写稿
- `.obsidian/`、`tmp/`、缓存目录和本地生成物
- `.env`、API key、token、证书、密码或其他凭据
- 含有个人身份、位置、联系方式或账户信息的文件

仓库默认使用 `.gitignore` 和公开文件 allowlist 阻止这些内容进入版本库。提交前可以运行：

```bash
bash tools/git/check-public-staged-files.sh
```

PR 和 `main` 分支推送会通过 GitHub Actions 再检查一次公开文件范围。

## 报告安全问题

如果你发现仓库中疑似包含私人资料、密钥、版权材料或其他不应公开的内容，请不要在公开 issue 中粘贴敏感正文。

推荐做法：

- 如果 GitHub 页面提供 `Security` -> `Report a vulnerability`，请通过该入口报告。
- 如果没有该入口，请只在公开 issue 中描述影响范围和文件路径，不要复制敏感内容。
- 如果问题涉及密钥泄露，应先停用或轮换相关密钥，再清理仓库历史。

## 维护者处理流程

处理公开安全问题时，优先顺序如下：

1. 立即删除或撤回误公开内容。
2. 如果涉及密钥，先轮换密钥，再处理 Git 历史。
3. 检查 `.gitignore`、`tools/git/check-public-staged-files.sh` 和 GitHub Actions 是否需要补充规则。
4. 通过普通 PR 流程记录修复，避免直接推送到 `main`。
